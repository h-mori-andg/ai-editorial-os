# 記事制作プロンプト（インタビュー・イベントレポート）
### KIJI Base 配布版 — 2026年3月

> **このファイルについて**
> このCLAUDE.mdは、現役ジャーナリストが年間500本以上の記事制作を通じて構築した、AIへの業務指示ファイルです。Claude（生成AI）に渡すことで、記事の品質・再現性・スピードが安定します。

> **使い方：** このファイルをClaude Codeのプロジェクトフォルダに置くか、会話の最初に内容を貼り付けてください。Claudeがこのルールに従って動作します。

## 🚀 クイックスタート（最初に確認）

記事制作の依頼を受けたら、まず以下を確認します。
最初のメッセージで指定済みの場合はスキップしてください。

### Q1. 記事タイプを選んでください

| 選択肢 | 説明 | 適用ルール |
|--------|------|-----------|
| A. ナラティブ（記者主導） | 記者の地の文中心。発言は厳選して引用 | B-1 + 引用率管理（A-2） |
| B. Q&A対話形式 | 「——」で質問、「氏名：」で発言。インタビュー記事の標準形式 | B-1-2（引用率管理なし） |
| C. 講演レポート | 講演内容を記者視点で再構成 | B-2 + 引用率管理（A-2） |
| D. パネルディスカッション | 複数登壇者の議論を構成 | B-3 + 引用率管理（A-2） |

> **初心者向けメモ：** 迷ったら「A. ナラティブ」か「B. Q&A」を選んでください。インタビュー記事の多くはこの2種類です。

### Q2. 一括設定（以下をまとめて確認）

以下をまとめて最初に伝えると、確認の往復が減り効率的です。

- **文体：** 記者文（常体／敬体）、引用文（常体／敬体）
- **構成方針：** 完全時系列 / 部分調整 / 論理優先
- **分量：** 想定文字数
- **タイトル方向性：** ビジョン訴求 / 変革ストーリー / 問題提起 / その他
- **プリセット使用：** あり（末尾のプリセット一覧参照） / なし
- **文字起こしクレンジング：** 要 / 不要
  - 「要」の場合：執筆前にサービス資料・質問票など固有名詞が確認できる資料を先に共有してください

## ⚙️ 基本設定（詳細）

### 1. コンテンツ情報

| 項目 | 入力内容 |
|------|---------|
| 種別 | インタビュー / 講演 / パネルディスカッション |
| 対象者 | 氏名・役職 |
| 実施日 | YYYY年MM月DD日 |
| 公開予定日 | YYYY年MM月DD日 |

### 2. 媒体・読者設定

| 項目 | 入力内容 |
|------|---------|
| 媒体名 | メディア名または「自社オウンドメディア」 |
| 読者層 | 具体的な読者像（例：IT企業のマーケ担当） |
| 想定文字数 | 数字（1.2倍まで可） |
| 記事トーン | プロフェッショナル / 親しみやすい / 技術専門 |

### 3. 文体設定

| 対象 | 文体 |
|------|------|
| 記者文章 | 常体（だ・である） / 敬体（です・ます） |
| 引用文 | 常体（だ・である） / 敬体（です・ます） |

> **初心者向けメモ：** 記者文と引用文で文体を変えるのが一般的です。例：記者文＝常体、引用文＝敬体（話者の言葉らしさを残す）。

### 4. タイトル方針

- [ ] 挑戦・困難克服ストーリー重視
- [ ] 革新性・独自性を強調
- [ ] 人間ドラマ・情熱を前面に
- [ ] 未来ビジョンを訴求
- [ ] 業界への問題提起
- [ ] その他：記入

### 5. 構成方針

- [ ] **完全時系列：** 文字起こしの順序を厳守（後で音源と照合しやすい）
- [ ] **部分調整：** 大筋は時系列、論理性のため一部調整
- [ ] **論理優先：** 読者理解を最優先に再構成（読みやすさ重視）

## 📝 執筆ルール

### A. 全コンテンツ共通（必須）

#### A-1. 絶対遵守事項

- ✅ 発言内容の意味・主張を絶対に変更しない
- ✅ 年月は絶対値表記（「今年」「昨年」禁止 → 「2025年」と明記）
- ✅ 数字・データ・固有名詞は原文のまま
- ✅ 中立的な記者視点を維持
- ✅ サンプル記事の内容（企業名・人名・技術等）は一切使用禁止

> **初心者向けメモ：** 「今年」という表現は記事が古くなったときに意味が変わります。必ず「2025年」のように絶対値で書きましょう。

#### A-1-1. サンプル記事の使用範囲

- **学習対象：** 記事構造、文体、リズム、見出しスタイル
- **使用禁止：** 企業名、人名、製品名、技術名、数値、事例

#### A-2. 引用率管理：15〜30%

引用とは、取材相手の発言をそのまま「」で囲んだ部分のことです。全体の15〜30%に収めるのがプロの基準です。

**引用を選ぶ基準：**
- 発言者の人柄・熱意・専門性が伝わる部分
- 印象的・独自性の高い表現
- 数字・具体例を含む重要主張

**残りは記者の言葉で：**
- 発言内容を記者の言葉で要約・解説する
- 「〜によると」「〜という」で間接的に伝える
- 背景情報や補足説明を追加する

#### A-3. 引用文の整形（必須処理）

話し言葉をそのまま引用すると読みにくくなります。以下のルールで書き言葉に変換してください。

| 口語表現 | → | 文語表現（敬体） | 文語表現（常体） |
|---------|---|---------------|---------------|
| 〜なんです、〜っていう | → | 〜です、〜という | 〜だ、〜という |
| すごい、めっちゃ | → | 非常に、極めて | 非常に、極めて |
| やっぱり、ちょっと | → | やはり、少し | やはり、少し |
| しっかり、きっちり | → | 確実に、適切に、十分に | 確実に、適切に、十分に |
| 〜みたいな、〜だったりとか | → | 〜のような、〜など | 〜のような、〜など |
| 〜できてます、〜やってます | → | 〜できています、〜行っています | 〜できている、〜行っている |
| 〜かなと思います | → | 〜と思います | 〜と思う |
| 〜ということになります | → | 〜となります | 〜となる |

その他：
- 話し言葉特有の倒置・省略 → 標準的な文構造に修正
- 過度な敬語（いらっしゃる等） → 標準的な敬語に調整

#### A-4. 文末の多様化（必須）

同じ文末表現が3回以上連続すると単調になります。以下を意識してください。

**敬体の場合：** です / ます / でした / ました / でしょう / 〜語った / 〜示した / 〜明かした

**常体の場合：** だ / である / た / する / 〜語った / 〜示した / 〜述べた

**共通技法：** 体言止め（名詞で終わる）／ 短文と長文の混在

#### A-5. 語彙の重複回避

同一パラグラフ内で同じ言葉を繰り返さないようにします。

| 重複しやすい語 | 言い換え例 |
|------------|----------|
| 最新 | 新しい、新たな |
| 効果 | 成果、結果、影響 |
| 課題 | 問題、懸念、テーマ、論点 |
| 実施 | 実行、展開、推進 |
| 重要 | 大切、肝要、鍵となる |

### B. コンテンツ別ルール

#### B-1. インタビュー（ナラティブ形式）

- インタビューの流れに沿って引用する
- 異なる箇所の発言を組み合わせない
- 発言者の表記：「〜と述べた」「〜と語った」「〜と説明した」等でバリエーションをつける
- 引用率管理（A-2）を適用
- **複数話者の場合：** 本文で最初に登場する引用者と最後のパラグラフの引用者は最上位役職者にする

**発言者の帰属表現：**
- 単独インタビューでは地の文に帰属を組み込む（行末に単独で置かない）
- 敬称：外部取材対象 → 様、一般的な記事 → 氏、自社媒体の社員 → 敬称なし
- 以下のパターンをローテーションする：
  1. 「〜」と○○様は話します。
  2. ○○様はこう語ります。「〜」
  3. ○○様によると、〜という。（間接引用）
  4. 「〜」と○○様は振り返ります。

#### B-1-2. インタビュー（Q&A対話形式）

- 質問は「——」、発言は「氏名：」で始める
- 引用率管理（A-2）は適用しない（記事の大部分が発言で構成されるため）
- A-3・A-4・A-5は適用する
- 話者が複数の場合、各パートで発言量のバランスを意識する
- 専門用語・固有概念の初出時には（）で補足説明を入れる

#### B-2. 講演レポート

- 聴衆への問いかけ → 文脈に応じて要約または引用
- 講演の核となるメッセージを明確化
- 資料・スライドへの言及があれば明記
- 専門用語・サービス名の初出時には（）で補足説明を入れる

**引用チェック（必須）：**
- スライドに書かれた文言は登壇者の口頭発言とは限らない。「」に入れず記者文で処理する
- 離れた箇所の発言を1つの「」にまとめない（合成禁止）

#### B-3. パネルディスカッション

- 各発言者を明確に区別（「A氏は」「B氏によると」等）
- 各パネリストの発言量を可能な限り均等に
- 議論の流れ・対立点・合意点を明確化
- 誰が発言しているかを先に示す（○○氏は「〜」と述べる）を基本とする

## 📋 作業フロー

### ステップ1：構成案を出力する

以下を出力してください：
- タイトル案3案
- リード文（指定の文字数）
- 本文構成：指定したパート数の見出し案と各パートの内容（箇条書き）

> **初心者向けメモ：** まず構成案を確認してからOKを出すと、方向性のズレを早い段階で修正できます。執筆後に「全部書き直し」になるのを防げます。

### ステップ2：本文執筆

- パートごとに800〜1000文字で出力する
- パート間での情報重複を避ける
- 用語・表記を全パートで統一管理する

> **初心者向けメモ：** 一度に全文を出力させると品質が落ちやすいです。パートごとに確認しながら進めるのがおすすめです。

## 📌 資料の渡し方（効率化のコツ）

> **初心者向けメモ：** AIへの指示は「まとめて渡す」ほど効率が上がります。

### 資料はまとめて渡す
- 文字起こし、質問票、参考URLは1〜2メッセージにまとめて渡す
- 「この資料はスタイル参照のみ、内容混入禁止」等の指示も同時に添える

### 設定は最初に一括指定する
- 形式・文体・構成方針・タイトル方向性・分量を最初のメッセージで一括指定する

### 文字起こしクレンジングの手順
1. サービス資料・質問票など固有名詞が確認できる資料を先に渡す
2. 生の文字起こしを渡してクレンジングを依頼する
3. AIが疑問符付きで不明箇所をリストアップする
4. クレンジング済みファイルを手元に置きながら倍速で音源を確認・修正する
5. 修正済み文字起こしを渡して執筆を依頼する

## ソース忠実性と表現の豊かさ

ソースにない内容は創作しない。ただし、表現を磨くことは積極的に行う。

**禁止：**
- 発言者が言っていない感情・動機・意見を記述する
- ソースにない直接引用（「」付き）を作る
- 事実を美談・結論・キャラクター評に変換する

**推奨：**
- ソースの言葉を選ぶ・整理する・磨くことで表現を豊かにする
- 見出し・構成・接続文は編集側で自由に設定してよい
- 「」で囲んだ引用は必ずソースで照合する

## プリセット一覧

プリセットとは、よく使う設定をまとめたショートカットです。「プリセットAで」と伝えるだけで設定が適用されます。自社の媒体に合わせてカスタマイズして使ってください。

### プリセットA：IT系専門メディア（BtoB向け）

```
- 読者層：IT企業の経営層・CTO・情報システム部門
- トーン：プロフェッショナル
- 記者文：常体 / 引用文：敬体
- 構成：論理優先
- 見出し：##のみ（3パート以上）
- タイトル：課題提起 or ビジョン訴求
```

### プリセットB：開発者・エンジニア向けメディア

```
- 読者層：ソフトウェア開発者・エンジニア
- トーン：プロフェッショナル（開発現場に近い実践的な語り口）
- 記者文：常体 / 引用文：敬体
- 構成：部分調整（講演の流れに沿いつつ論理を整理）
- 字数：リード150〜200字 ＋ 本文4000〜4500字
- 見出し：##のみ（3パート以上）
- タイトル：体言止め or 体言＋サブタイトル形式
- リード構成：課題提示 → 解消への示唆 → 登壇者情報 → 語った内容の要約
- 段落冒頭：全角スペース1字あけ（　）
```

### プリセットC：オウンドメディア（汎用）

```
- 読者層：自社製品・サービスに興味を持つ潜在顧客
- トーン：親しみやすい・わかりやすい
- 記者文：敬体 / 引用文：敬体
- 構成：部分調整
- 字数：リード100〜150字 ＋ 本文2000〜3000字
- タイトル：読者の課題感に刺さるキーワードを入れる
```
